Where NCTSSoS monomials burn memory — and the three-tier fix

One Pauli monomial product costs ~10 heap allocations (432 B); an 80×80 moment-matrix loop allocates 36 MB of pure transients. A measured prototype removes 3.7–10× of that without changing the data representation.

Why this matters: moment/SOS relaxation assembly is a tight loop of word concat → simplify → dict lookup over hundreds of thousands of basis pairs. Every wasted allocation there is multiplied by n²·k — it shows up directly as GC time and cache misses at problem scale.

Triage update (2026-06-12, branch perf/monomial-allocations): end-to-end solves are solver-dominated (NCTSSoS ≈0.1% of a COSMO run), but the build phase is 94% NCTSSoS-owned allocation — 8 GB churn / 29% GC at N=18, with moment_relax as the dominant phase and a measured ≥96% interning hit rate. Verdict: Tier 1 go now (with non-mutating public constructors + property tests), Tier 2 go as a profile-directed sprint after Tier 1 (gates: N=18 ≤ 2 GB, GC < 10%, ≥2× wall), Tier 3 deferred behind explicit triggers. Full detail in Triage & Roadmap below.

Quick Overview

TL;DR. The monomial type itself (NormalMonomial{A,T}, an immutable word Vector{T}) is fine for correctness — the problem is the plumbing around it. Every product funnels a provably single-term result through a full Polynomial construction with sorting; simplify copies buffers the caller already owns; m * p accumulates quadratically; and Julia's generic array hash allocates on every dict probe. A Tier-1 prototype (verified bit-identical results) gets 3.7–10× with localized edits to 4 files.

Core Concepts

A monomial is a word. NormalMonomial{A,T} stores a product of generators as an integer vector — e.g. UInt8[1, 8, 15] means σx₁·σy₃·σz₅ in the Pauli encoding. The algebra A lives in the type parameter, so dispatch is free; the word lives on the heap, so everything else isn't.

"Normal" means canonical. The constructor enforces that the word is already in the algebra's normal form (sorted by site, no squares, etc.) by running _validate_word — an O(n) scan — on every construction. That's the right call at the API boundary and a pure tax when the word just came out of simplify, which guarantees canonical output by construction.

Multiplication = concatenate, then rewrite. m₁ * m₂ concatenates the two words and calls the algebra's rewrite rules (simplify). For Monoid and TwistedGroup algebras (NonCommutative, Projector, Unipotent, Pauli) the result is always zero or one term. Only PBW algebras (Fermionic, Bosonic) can genuinely expand into many terms.

Polynomials are sorted-unique term lists. Polynomial{A,T,C} keeps Vector{Tuple{C, NormalMonomial}} sorted with unique monomials and nonzero coefficients. Maintaining that invariant requires a sort + dedup pass (_process_terms) — which today runs even for single-term inputs.

The hot loop. Moment-matrix assembly computes simplify(A, aᵀ·m·b) for every basis pair (a, b) and Hamiltonian term m, then uses the canonical word as a dictionary key. Cost per entry ≈ buffer alloc + rewrite + hash + probe. Multiply by basis² × terms.

Cause 3 in code: the quadratic accumulator

From src/types/arithmetic.jl, m::NormalMonomial * p::Polynomial. Each iteration rebuilds, converts and re-sorts the entire accumulated polynomial — ~2.8 KB per term where ~190 B suffices.

for (coef, mono) in p.terms
    iszero(coef) && continue
    prod_poly = m * mono              # full Polynomial machinery per term (~416 B)
    result = result + coef * prod_poly  # vcat + convert + FULL re-sort, every time
end

Cause 2 in code: the wasted copy in the hottest loop

From src/optimization/moment.jl (≈ lines 471, 535, 555, 576): _neat_dot3 returns a fresh buffer nobody else owns — then simplify copies it again before mutating. Measured waste: ~40 ns + 80 B per moment entry.

The Tier-1 kernel (prototyped, results bit-identical)

# single-term multiply kernel: Monoid / TwistedGroup algebras only
function _mul_term(a::NormalMonomial{A,T}, b::NormalMonomial{A,T}) where {A,T}
    buf = Vector{T}(undef, length(a.word) + length(b.word))   # the ONLY buffer
    copyto!(buf, a.word)
    copyto!(buf, length(a.word) + 1, b.word, 1, length(b.word))
    word, phase = simplify!(A, buf)                # in place — no defensive copy
    return _phase_coef(phase), NormalMonomial{A,T}(word, _trusted)  # skip validation
end

# m * p: one pre-sized buffer, ONE Polynomial() at the end
function Base.:(*)(m::NormalMonomial{A,T}, p::Polynomial{A,T,C}) where {A,T,C}
    out = Vector{Tuple{C,NormalMonomial{A,T}}}()
    sizehint!(out, length(p.terms))
    for (coef, mono) in p.terms
        c, mm = _mul_term(m, mono)
        push!(out, (coef * c, mm))
    end
    return Polynomial(out)        # single sort + dedup, at the boundary
end

The three tiers

  • Trusted constructor — a private path skipping any(iszero) + _validate_word for simplify-derived words. Validate at the boundary, trust inside.
  • Single-term kernel _mul_term for Monoid/TwistedGroup algebras — no terms-Vector, no Polynomial, no sort. PBW keeps the multi-term path.
  • Buffered m*p, p*m, p*p — push into one pre-sized vector, one Polynomial() at the end.
  • _process_terms: sort! in place; skip entirely for ≤ 1 term.
  • simplify! adoption in arithmetic.jl + 4 moment.jl call sites.
  • Allocation-free hash(::NormalMonomial) — chunk-wise hashing of word bytes instead of Julia's generic AbstractArray hash (16 B/probe today).

Measured: m*m 3.7× / 4.5× mem · m*p 10× / 15× mem · p*p 1.5× / 2.4× mem. Existing test suite is the regression net.

  • _neat_dot3!(scratch, a, m, b) — write the triple product into one reusable scratch buffer per assembly loop (thread-local if parallel), then simplify! in place.
  • Intern canonical words — get! into a Dict{Vector{T},NormalMonomial}, copying the scratch only on first sight; or key the moment-index dict directly by word.
  • Effect — steady-state assembly allocates only for novel canonical words (the set saturates fast): tens of KB instead of tens of MB; kills the 2.4% GC overhead.
  • Option A — inline small words. isbits small-buffer storage (SmallVector{N,T} or NTuple+len, N≈16). Makes ==/hash/isless register ops, removes both heap objects per monomial. Cost: capacity parameter or spill-to-heap hybrid; churn across Polynomial, states, composed monomials. General win.
  • Option B — symplectic Pauli words. (x-mask, z-mask, phase) bit-vectors; products = XOR + popcount, O(1) vs sort-based O(n log n). The standard stabilizer trick (cf. QuantumClifford.jl). Can stay an internal assembly representation.
  • Gate: only after Tiers 1–2, only if profiles still show monomial plumbing.

Rejected: global hash-consing (thread-safety burden for pointer-equality wins Tier 2 already gets locally); lazy/unsorted Polynomial (the sorted-unique invariant is load-bearing).

Decisions

Tier 1 — GO NOW

  • Public Polynomial(::Vector) stays non-mutating; in-place sort! only via internal owned-buffer path
  • Trusted ctor stays private; per-algebra simplify! postconditions documented; never wrap a scratch buffer without copy
  • Hash contract: algebra type + seed h, isequal consistency, cross-algebra collision tests
  • Differential property tests across all algebras (random words, duplicate-heavy, zeros, PBW expansions)

Tier 2 — GO, AFTER TIER 1

  • Profile-directed sprint: line-level allocation table of moment_relax before implementing
  • Never store the scratch as a Dict key — probe by scratch, copy on first sight
  • Track peak RSS, not just churn (interning retains memory)
  • Gates: N=18 build ≤ 2 GB churn · GC < 10% · ≥2× wall · peak RSS not worse

Tier 3 — DEFERRED, WITH TRIGGERS

  • Re-profile after Tier 2 with the production solver (Mosek)
  • Trigger a representation spike if word storage/hash/== still > ~25% of build allocations
  • …or if N≥30 Pauli workloads remain build-bound
  • Symplectic Pauli (Option B) = only order-of-magnitude candidate; scope as internal assembly representation

Verdict on “should we push for even less allocation?”
Yes for the build path, no for the representation rewrite — yet. End-to-end wall clock is solver-dominated, so zero-allocation as a goal in itself buys nothing visible there. But the build phase is 94% NCTSSoS-owned with GC% climbing in N — real headroom and a real scaling risk. Tiers 1+2 attack exactly that with measured justification; Tier 3 stays behind explicit triggers.

Evidence limits (kept honest)
  • The end-to-end ratio is COSMO-specific and N=6-sized; Mosek shrinks solve time and shifts the ratio toward build — re-measure before any Tier 3 call.
  • Allocation churn ≠ peak RSS; the 8 GB figure is GC pressure, not footprint.
  • Profile sample attribution misses BLAS threads.
  • Interning saturation measured on Heisenberg-type supports; check one non-chain Hamiltonian during the Tier 2 sprint.

Common Pitfalls & Gotchas

▸ The inner Polynomial constructor lies in its docstring

It claims to enforce "sorted, deduplicated, non-zero coefficients" — it stores terms as-given. It's actually the trusted path (the outer constructor processes). Fix the comment to state the real contract before someone "fixes" the code the wrong way.

▸ A dead outer constructor

Two outer Polynomial(::Vector{Tuple{C,NormalMonomial}}) methods exist; the unconstrained-CIn one (which skips _process_terms) is unreachable for valid coefficient types because the struct bounds C<:Number and the constrained method is more specific. Delete it.

▸ "Accumulate by +" is quadratic on sorted containers

result = result + term on a sorted-invariant polynomial re-sorts everything per step. Any new loop touching polynomials should push into a raw buffer and construct once. This pattern will creep back in via review unless the buffered helpers are the obvious, documented way.

▸ Don't delete validation — relocate it

The trusted constructor must stay private. _validate_word on user-facing constructors is what makes "a NormalMonomial is always canonical" a real invariant instead of a hope. The win comes from not re-checking words that simplify just produced.

▸ Micro-benchmarks of unused results lie

The bare constructor benchmarked at 9.8 ns while its validation alone took 59 ns — the compiler hoisted the loop-invariant call. Trust the allocation counts and the end-to-end loop numbers (36 MB doesn't lie); treat ns-level micro-numbers as directional.

▸ Julia 1.12 generic array hashing allocates

hash(::Vector{UInt8}) = ~22 ns + 16 B. Invisible in isolation, but moment assembly does millions of dict probes. A custom word hash is ~10 lines and removes it entirely.

Test your understanding

1. Why is it safe for the trusted constructor to skip _validate_word?

Because its only callers pass words produced by simplify/simplify!, whose postcondition is canonical form. Validation at that point re-proves a theorem the simplifier just proved. User-facing constructors keep validating.

2. Why does m * p improve 10× but p * p only 1.5×?

p * p already pushed all products into one buffer with a single final sort — its waste was per-product plumbing (copy, terms-Vector). m * p had the additional quadratic accumulator (result = result + ... re-sorting per term), so fixing the loop structure dominates.

3. Which algebras can use the single-term kernel, and which can't?

Monoid (NonCommutative, Projector, Unipotent) and TwistedGroup (Pauli) products always yield ≤ 1 term — kernel applies. PBW algebras (Fermionic, Bosonic) genuinely expand into multiple terms via commutation relations — they keep the multi-term path.

4. What makes Tier 2's interning idea cheap where global hash-consing was rejected?

The intern table is local to one assembly loop — no global mutable state, no thread-safety contract, dropped when assembly ends. It exploits the same fact (canonical words repeat heavily) without the lifecycle burden.
