## Per-Feature Details

### Pauli Algebra End-to-End (Beginner Mei)

- **Role:** First-year QI grad student, first SDP, follows docs literally
- **Use Case:** Zero to solved problem using quick-start + Pauli examples
- **What they tried:** N=4 and N=6 Heisenberg chains with COSMO, orders 1 and 2, with/without term sparsity
- **Discoverability:** Good. The two literate examples (`pauli_algebra_interface.jl`, `ground_state_energy.jl`) provide a clear copy-paste path. Docstrings for `polyopt`, `create_pauli_variables`, `SolverConfig`, `cs_nctssos` are thorough.
- **Setup:** `Pkg.instantiate()` worked. COSMO available as test dependency.
- **Functionality:**
  - `create_pauli_variables(1:N)` works perfectly
  - Integer coefficients auto-promote to ComplexF64 (no `ArgumentError` -- **better** than expected)
  - `polyopt`, `cs_nctssos`, `cs_nctssos_higher` all work
  - N=4 order=2: E/site = -0.500 (exact); N=6 order=2: E/site = -0.4671 (matches docs)
- **Expected vs Actual:** All expectations met or exceeded. Integer coefficient auto-promotion was a pleasant surprise.
- **Blocked steps:** None
- **Friction points:**
  1. **Polynomial display prints raw `UInt8` arrays** (`UInt8[0x01, 0x04]` instead of `sigma_x1 * sigma_x2`). Cannot verify Hamiltonian correctness visually. **Highest-impact UX issue across all tests.**
  2. All examples hardcode `MosekTools`/`Mosek.Optimizer`. No COSMO or Clarabel alternative shown.
  3. `MMD()`, `MF()`, `MaximalElimination()` listed as options but not explained.
  4. `cs_nctssos_higher` purpose and "when to call it" not explained.
  5. COSMO is test-only dep; `Pkg.add("NCTSSoS")` alone won't provide any solver.
- **Doc suggestions:** Add COSMO examples, fix monomial display, add elimination algorithm glossary, explain `cs_nctssos_higher` purpose.

### Sparsity Exploitation & Benchmarking (Intermediate Rohan)

- **Role:** Postdoc, fluent in Julia/JuMP, migrating from TSSOS/ncpol2sdpa
- **Use Case:** Exploit sparsity, mix algebra types, benchmark against known tools
- **What they tried:** CHSH with all 4 elimination algorithms, `cs_nctssos_higher` on Example 3.4, `ComposedMonomial` with Pauli+Fermionic, custom `moment_basis`
- **Discoverability:** Partial. Literate examples are well-structured, but sign convention inconsistency and missing export docs cause confusion.
- **Setup:** Worked. COSMO substitution straightforward.
- **Functionality:**
  - CHSH: `objective = -2.8284` (correct, matches -2sqrt(2) to 1e-10)
  - 4 elimination algorithms: MF/MMD give 2 cliques, MaximalElimination/NoElimination give 1. All bounds consistent.
  - **AsIsElimination returns `-4.0` (invalid bound)** on non-chordal CHSH graph. No runtime error/warning.
  - `cs_nctssos_higher`: correctly demonstrates graph stabilization without bound exactness (Example 3.4 gap = 0.0026)
  - `ComposedMonomial((pauli_mono, fermionic_mono))` works without type errors
  - Custom `moment_basis` in `SolverConfig` works correctly
- **Expected vs Actual:**
  - CHSH sign: docs say `result.objective ~ +2sqrt(2)`, actual is `-2sqrt(2)`. **Doc bug.**
  - `cs_nctssos_higher` did not improve (correct for Example 3.4 -- stabilization trap confirmed)
  - All other expectations met
- **Blocked steps:** None
- **Friction points:**
  1. **CHSH sign convention wrong in `bell.jl`** (line 111-118). Says `abs(chsh_bound - tsirelson_bound)` should be ~1e-7, but `chsh_bound ~ -2.828` and `tsirelson_bound = +2.828`, so the difference is ~5.66. **Critical doc bug.**
  2. **`simplify` export collision with JuMP.** `using NCTSSoS, JuMP` then `simplify(cm)` fails with `UndefVarError`. Requires `NCTSSoS.simplify(cm)`.
  3. `AsIsElimination` silently produces invalid bounds on non-chordal graphs (no runtime warning).
  4. Monomial display is opaque (same as beginner report).
  5. No TSSOS migration guide or keyword aliases.
  6. `edge_pairs` and `chordal_completion_edges` from sparsity examples are not exported.
- **Doc suggestions:** Fix CHSH sign convention, document `simplify` collision, add elimination algorithm comparison table, add TSSOS migration guide, export or document graph utility functions.

### GNS Extraction Edge Cases (Expert Dr. Lena)

- **Role:** Senior operator algebra researcher, reads source code, tests mathematical guarantees
- **Use Case:** Push GNS extraction edge cases -- degenerate optima, flatness detection, numerical tolerances
- **What they tried:** BKP Example 4.23 (XYX on polydisc), SVD vs Cholesky comparison, all 3 `gns_reconstruct` overloads, flatness testing, robustness diagnostics, missing moment entries, varying atol (1e-4 to 1e-12)
- **Discoverability:** Partial. GNS docs are thorough but `monomap` key type is undocumented, `H_deg` vs relaxation `order` mapping left implicit, `flat_extend` as mandatory pre-step is buried.
- **Setup:** Worked cleanly.
- **Functionality:**
  - BKP Example 4.23: objective = -1.0, GNS rank = 5, min eigenvalue = -1.0, moment max error = 1.1e-8. **All correct.**
  - SVD vs Cholesky: both rank 5, max moment difference = 3.2e-9. **Excellent agreement.**
  - Overload 1 (Hankel) vs Overload 2 (monomap): identical results (diff = 0.0).
  - Flatness: raw Hankel NOT flat (rank 5 vs 9). After `flat_extend`: exactly flat (rank 5 = 5).
  - `robustness_report`: sigma_min=0.022, condition=93.4, dist_to_flat=0.035. Reasonable.
  - `verify_gns`: symmetric, moment error 1.1e-8, objective error 1.1e-8, constraints feasible.
  - Missing moments: **structured `ArgumentError`** listing first 5 missing keys. **No cryptic KeyError. Good.**
  - atol=1e-12: rank inflates from 5 to 7 (picks up noise ~6e-9). **Graceful degradation.**
  - Identity monomial always first in `gns.basis`. **Invariant holds.**
- **Expected vs Actual:** All expectations met. No bugs found.
- **Blocked steps:** Sparse GNS (overload 3) not tested -- polydisc has one clique; existing test suite covers this.
- **Friction points:**
  1. **`flat_extend` is silently critical.** Skipping it with `:cholesky` produces a warning but still returns output that may have incorrect rank. Not flagged as mandatory in docs.
  2. `monomap` key type (`symmetric_canon(expval(mono))`) undocumented. Had to read `moment.jl` source.
  3. `H_deg` vs relaxation `order` relationship not stated as a rule (use `H_deg=order`, `hankel_deg=order-1`).
  4. `robustness_report` requires keeping Hankel matrix separately (not stored in GNSResult).
  5. `atol=1e-12` danger not documented (inflates rank with solver noise).
- **Doc suggestions:** Add "Recommended Workflow" section with `flat_extend` as standard step, document monomap key type, add H_deg/order mapping rule, warn about atol below solver noise floor.

## Issues Found

### Critical (3)

1. **CHSH sign convention bug in `bell.jl`** (line 111-118). The example compares `result.objective` (which is negative, ~-2.828) against `+2*sqrt(2)`, producing a difference of ~5.66 instead of the claimed ~1e-7. Either negate the objective in the example or compare against `-2*sqrt(2)`. This will mislead every Bell inequality user.

2. **Polynomial/monomial display prints raw `UInt8` byte arrays** instead of human-readable symbol names. Users cannot visually verify their problem setup. This affects all experience levels and is the single biggest UX gap in the library. Both Mei (beginner) and Rohan (intermediate) independently flagged this.

3. **`simplify` export collision with JuMP.** Since most NCTSSoS users will `using NCTSSoS, JuMP` (or the solver pulls JuMP in), calling `simplify(...)` fails with `UndefVarError`. Users must qualify as `NCTSSoS.simplify(...)`, which is undocumented.

### High

4. **`flat_extend` as mandatory GNS pre-step is not documented.** Raw solver Hankel matrices are typically NOT flat. Passing them to `gns_reconstruct` with `:cholesky` produces a warning but still returns potentially inaccurate output. The recommended workflow should be explicit.

5. **`AsIsElimination` silently produces invalid bounds** on non-chordal graphs (-4.0 for CHSH instead of -2.828). No runtime error or warning. A practitioner comparing all algorithms would be confused.

6. **All examples assume Mosek.** No open-source solver (COSMO, Clarabel) is shown. Users without a Mosek license have no documented path.

### Medium

7. **`monomap` key type undocumented.** Users must read `moment.jl` source to learn keys are `symmetric_canon(expval(mono))`.
8. **`H_deg` vs relaxation `order` relationship** left implicit. Should state: use `H_deg=order`, `hankel_deg=order-1`.
9. **Elimination algorithm glossary missing.** `MF`, `MMD`, `MaximalElimination`, `NoElimination` not explained.
10. **`cs_nctssos_higher` purpose and when-to-call guidance missing.**
11. **`edge_pairs`, `chordal_completion_edges` used in examples but not exported.**
12. **No TSSOS migration guide** for users switching from TSSOS/ncpol2sdpa.
13. **`atol` below solver noise floor** inflates GNS rank; not warned in docs.

## Suggestions

Ordered by impact:

1. **Fix monomial `show` method** to use registry symbol names. This is the highest-ROI change for all user levels.
2. **Fix CHSH sign convention** in `bell.jl`. Either negate objective or compare against `-2sqrt(2)`.
3. **Resolve `simplify` export collision.** Options: stop exporting `simplify`, rename to `simplify_poly`, or document the workaround prominently.
4. **Add COSMO examples** alongside Mosek in at least the quick-start pages.
5. **Document the GNS recommended workflow** with `flat_extend` as standard pre-step.
6. **Add runtime warning for `AsIsElimination`** when the graph is not chordal, or throw an error.
7. **Add elimination algorithm comparison table** to docs.
8. **Document `monomap` key type** and `H_deg`/`order` mapping rule.
9. **Add TSSOS migration guide** for the target audience.
10. **Add `atol` guidance** in GNS docstrings (default 1e-8 is well-chosen; going below ~1e-9 risks noise).
