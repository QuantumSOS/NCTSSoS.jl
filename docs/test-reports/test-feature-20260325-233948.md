## Expected vs Actual Outcome

Most edge cases handled well: empty Pauli/Fermionic/Bosonic inputs return empty registries, large N auto-selects correct index types, unicode prefixes work, negative subscripts rejected clearly. The gaps are in prefix-based functions (empty input crash) and duplicate detection (opaque errors).

## Issues Found

### Bug (1)

1. **Empty `prefix_subscripts` crashes prefix-based functions.** `create_projector_variables(Tuple{String,Vector{Int}}[])`, `create_unipotent_variables(...)`, and `create_noncommutative_variables(...)` all throw `ArgumentError("reducing over an empty collection is not allowed")` from `sum(x -> length(x[2]), prefix_subscripts)` in `_create_noncommutative_variables` (registry.jl:663). Pauli/Fermionic/Bosonic handle empty inputs gracefully. Fix: add `init=0` to the `sum()` call.

### Missing validation (2)

2. **Duplicate subscripts produce opaque error.** `create_pauli_variables([1,1,2])` fails with `"Inconsistent registry: different lengths"` — the Dict loses entries from duplicate symbols. Same for all 6 functions. The user gets no hint about duplicates. Fix: detect duplicates early in each `create_*` function.

3. **Duplicate subscripts across same-named prefix groups also opaque.** `create_projector_variables([("P", [1,2]), ("P", [1,2])])` fails with the same unhelpful message. Fix: detect symbol collisions before constructing the registry.

### Rough edges (5)

4. **Empty prefix string silently accepted.** `create_noncommutative_variables([("", 1:2)])` produces symbols `Symbol("₁")`, `Symbol("₂")` — bare subscripts with no prefix. Works but confusing.

5. **`_subscript_string` only accepts `Int`.** Typed as `(n::Int)`, rejects `Int32`, `UInt8` directly. Currently not triggered by public API but fragile.

6. **Non-integer subscripts for Pauli/Fermionic/Bosonic give `MethodError` on `_subscript_string`.** Accurate but unfriendly. Prefix-based functions catch this earlier via type constraints.

7. **Display registry is last-writer-wins.** Creating two Pauli registries overwrites the global display fallback. First registry's monomials may display with wrong names. Documented limitation.

8. **Subscript 0 allowed.** `create_pauli_variables([0])` → `σx₀, σy₀, σz₀`. Fine mathematically but may surprise 1-based-indexing users.

| Category | Result |
|---|---|
| Empty inputs (Pauli/Fermionic/Bosonic) | 0-length registry, empty vectors |
| Empty subscript range in prefix group | `[("P", 1:0)]` → 0-length group |
| Single element | All 6 correct |
| Large N (100–10000) | Correct; auto type selection (UInt8→UInt16); 10k Pauli in ~0.2s |
| Non-contiguous subscripts | `[1,3,5,7]` works; labels use subscript value |
| Negative subscripts | Clear error message |
| Unicode prefixes (`α`, `β`, emoji) | All work |

| Cross-algebra equality | `c[1] == a[1]` is `false` (correct) |
| StepRange, reverse range | Both work |

## Suggestions

1. **Fix BUG-1:** Add `init=0` to `sum()` in `_create_noncommutative_variables` (1-line fix)
2. **Fix MISSING-1/2:** Add duplicate-subscript/symbol detection early in `create_*` functions
3. **Consider:** Reject or warn on empty-string prefixes
4. **Consider:** Widen `_subscript_string` to accept `Integer` instead of `Int`
