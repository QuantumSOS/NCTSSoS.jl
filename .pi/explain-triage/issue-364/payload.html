Evidence

Why the current code can't handle N=100 at degree 4

Four hard-coded barriers in src/optimization/symmetry.jl:

Barrier Location What it blocks
max_degree ∈ 0:2 validation Lines ~1021, ~2776 Any charge sector analysis above degree 2

Why the architecture is ready for extension

The internal machinery already supports the needed features — only validation guards and input constructors are missing:

1

Orbit reducer handles phases

At line ~1948, _build_orbit_reducer tracks orbit_phase[image_sym] and detects sign conflicts via a zero_orbit flag. Sign symmetry is structurally supported — it just can't reach this code.

2

Basis closure check is phase-aware

At line ~1924, _check_basis_closure checks that the monomial image is in the basis set. For sign symmetry the monomial maps to itself (with a sign), so it's still in the basis. The sign is handled separately by the orbit reducer's phase dictionary.

3

Charge-word/basis size equality holds for contiguous chains

For a contiguous chain basis of degree \(d\) with \(N\) sites, the number of basis monomials \(\left(1 + N \cdot \sum_{\ell=1}^{d} 3^\ell\right)\) exactly equals the number of basis-driven charge words. The square-transform assumption in _pauli_charge_transform_groups (line ~3049) is not violated.

What "contiguous chain basis" means

In quantum spin systems, a Pauli monomial is a product of Pauli operators (\(\sigma^x, \sigma^y, \sigma^z\)) acting on specific qubit sites. The "full NPA basis" at degree \(d\) includes all monomials on any subset of up to \(d\) sites out of \(N\) — yielding \(O(N^d)\) elements.

A contiguous chain basis restricts to monomials where the sites form a contiguous block on the lattice (with periodic wrapping for a ring geometry). For a 1D periodic chain:

This is physically motivated: the Heisenberg Hamiltonian has only nearest-neighbor interactions, so a contiguous-support basis captures the relevant correlations without combinatorial blowup.

Sign Symmetry Support

~30 lines changed

What: Add pauli_sign_symmetry(N) constructor and relax validation in _validate_pauli_spatial_group.

Why this route: The orbit reducer already handles phase accumulation and sign-conflict detection. Only the validation layer needs relaxation — "must be a pure site permutation" → "must map each charge word to ±1 multiple of another charge word with the same charge."

Key subtlety

Under the sign symmetry generator (conjugation by \(\prod_i \sigma_i^z\)), a Pauli X or Y picks up −1 while Z is unchanged. A monomial with \(k\) occurrences of X or Y gets eigenvalue \((-1)^k\). Monomials with different X+Y parity live in different eigenspaces — the orbit reducer handles this correctly via zero_orbit detection.

Files changed

  • src/optimization/pauli_chains.jl (add pauli_sign_symmetry)

Caveats & Open Questions

Q1: Should the chain basis support open (non-periodic) chains?

The triage plan implements periodic chains first. Open boundary conditions change the size formula slightly (boundary sites have fewer neighbors) but are described as a ~5-line extension. No decision has been made on whether to include this in the initial PRs.

Q2: What if a user provides an arbitrary sparse basis that isn't "charge-complete"?

Current code assumes length(charge_words) == length(basis) (square transform). The plan chooses Option A (conservative): require custom bases to be "charge-complete." Error otherwise.

Option B (rectangular transforms) would require nontrivial Wedderburn changes and is deferred.

Q3: Will N=100 d=4 symbolic decomposition complete in <120s?

Estimate: 12,000 charge words × 400 group elements = 4.8M operations. Should be feasible but hasn't been profiled. A timeout annotation is recommended. If it exceeds 120s on CI, mark as @test_skip.

Q4: Should PauliSingletConstraintSpec.max_degree cap also be lifted?

The existing singlet constraint path (_add_pauli_singlet_constraints!) only handles degree 1-2. Recommendation: lift cap to 4 only when PR 4 (singlet reducer) is implemented, keeping cap at 2 for the constraint-emission path until then.

Risk: Periodic wrapping in charge-word generation

Site-subsets like {99, 100, 1, 2} on a 100-site ring must be treated as a valid contiguous-support subset. The charge-word extractor should use the monomial's actual site set (as produced by the chain basis generator), not infer contiguity from monotone ordering. This requires specific test coverage.

Risk: Sign symmetry composition with spatial orbits

If two monomials with different X+Y parity fall in the same spatial orbit (same site structure under translation/reflection), the orbit reducer would detect a sign conflict and zero the orbit. The triage document asserts this doesn't happen for contiguous chains but recommends defensive testing.
