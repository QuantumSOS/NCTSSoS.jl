# StatePolyOpt Solver Support

## Task Status: PARTIALLY COMPLETE

## Summary

StatePolyOpt solver support is implemented and working for most use cases. Tests 7.2.0 and 7.2.2 pass. Test 7.2.1 is a known limitation documented below.

## Working Features

1. **StatePolyOpt type**: `polyopt(ncsp, registry)` creates state polynomial optimization problems
2. **Solver interface**: `cs_nctssos(spop, solver_config)` solves state polynomial optimization
3. **Basic state polynomials**: Objectives with single expectations like `<xy>` work correctly
4. **Covariance expressions**: Test 7.2.2 with `cov(a,b) = <xy> - <x><y>` works

## Known Limitation: Test 7.2.1

### Problem Description
Test 7.2.1 has an objective with **squared expectations**:
```julia
sp1 = ς(x[1] * y[2]) + ς(x[2] * y[1])  # <xy> + <wz>
sp = -sp1 * sp1 - sp2 * sp2            # includes <xy><xy> terms
```

The solver returns ~0 instead of the expected -4.0.

### Root Cause Analysis

1. **Basis limitation**: `get_state_basis` generates NCStateWords with identity StateWord part: `<I>*M`

2. **_neat_dot3 behavior**: When computing moment matrix entries:
   - `_neat_dot3(<I>*a, I, <I>*b)` produces `<I>*(a†b)` - a single expectation
   - Cannot generate compound StateWords like `<A><B>` (products of expectations)

3. **Missing terms**: The objective has terms `<xy><xy>` which require compound StateWords with 2 state_monos. These are NOT generated by the current basis.

4. **Consistent relaxation**: Both objective coefficients and constraint matrices ignore terms not in the basis, resulting in a relaxed problem that doesn't capture the full objective.

### Attempted Fix (Reverted)

Added `<M>*I` form NCStateWords to the basis alongside `<I>*M` forms:
```julia
# Form 1: <I> * M (current)
ncsw = NCStateWord(identity_sw, term.monomial)
# Form 2: <M> * I (added)
ncsw_sw = NCStateWord(StateWord{ST}(term.monomial), identity_mono)
```

**Result**: This DID generate compound StateWords via `_neat_dot3(<A>*I, I, <B>*I) = <A><B>*I`.

**Problem**: Caused double-counting because:
- Both `<M>*I` and `<I>*M` have the same `expval` = `<M>`
- But `_neat_dot3(<M>*I, I, <M>*I)` produces `<M><M>` (compound)
- While `_neat_dot3(<I>*M, I, <I>*M)` produces `<I>*(M†M)` (simplified to identity)
- This creates inconsistent constraints, giving -8 instead of -4

### Why Test 7.2.2 Works

Test 7.2.2 has compound terms `<x><y>` from `ς(x[a]) * ς(y[b])`, but they are **products of different expectations**. The optimization happens to give the correct answer despite missing these terms, likely because:
1. The optimal solution sets these product terms to values consistent with PSD constraints
2. The single-expectation terms `<xy>` fully determine the optimal value

### Proposed Solutions

1. **Use ONLY `<M>*I` forms** for objectives with squared expectations (different basis mode)

2. **Create a separate compound basis**: Generate NCStateWords with multi-term StateWords directly for objectives that need them

3. **StateWord-indexed moment matrix**: Change the moment matrix to be indexed by StateWords, not NCStateWords, with proper handling of compound terms

4. **Objective-aware basis**: Analyze the objective to determine which NCStateWord forms are needed and generate only those

### Known Issue: Term Sparsity (MMD) with State Polynomials

The MMD term sparsity algorithm creates incorrect blocks for state polynomial optimization.
When enabled, it returns 0 instead of the correct value. This is because the term sparsity
graph construction doesn't properly handle NCStateWord/StateWord types.

Workaround: Use `ts_algo=NoElimination()` for state polynomial optimization.

## Next Steps

To fully support test 7.2.1 (compound expectations like `<A>²`):

### Why This Is Hard

The fundamental issue is mathematical, not just implementation:

1. **Current approach**: Uses NCStateWord-indexed moment matrix with `<I>*M` basis
   - Moment matrix entries: `M[i,j] = <basis_i† * op * basis_j>` = single expectation
   - Cannot represent `<A><B>` (product of expectations) in this framework

2. **The `<A>²` problem**: The term `<xy>²` is the SQUARE of an expectation value
   - This is NOT the same as `<(xy)²>` = `<xyxy>`
   - `<xy>²` can range from 0 to 1 (square of value in [-1,1])
   - `<xyxy>` can be negative (e.g., for anticommuting operators)

3. **Attempted fix failed**: Using `<M>*I` basis to generate `<A><B>` works but:
   - Breaks the moment matrix structure (double-counting variables)
   - Results in infeasible or incorrect problems

### Fundamental Redesign Needed

To support compound expectations properly:

1. **StateWord-indexed variables**: Moment variables indexed by StateWords, not NCStateWords
   - Variable for `<xy>` separate from variables for `<x>`, `<y>`
   - Variable for `<xy>²` = `(<xy>)²` separate from variable for `<xyxy>`

2. **Polynomial constraints**: Add constraints like `<xy>² = <xy>²` (trivial) 
   and enforce consistency with PSD structure differently

3. **Different SDP formulation**: May need to use different constraint structure
   that captures products of expectations

This is a significant architectural change beyond the current StatePolyOpt implementation.

### Workarounds

For now, users with squared expectations should:
1. Reformulate the problem to avoid `<A>²` terms if possible
2. Use higher-level relaxations that may give correct bounds
3. Wait for architectural improvements

See `research.md` for original implementation plan.
