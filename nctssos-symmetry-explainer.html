Symmetry Reduction in NCTSSoS.jl

How the library exploits the symmetry of your optimization problem to shrink the SDP the solver actually sees — without touching the rest of the package.

Why this matters: A symmetry-adapted basis can turn one huge PSD constraint into several tiny ones. For the CHSH Bell inequality, a 5×5 moment matrix becomes three independent 1×1 blocks — same answer, orders-of-magnitude less solver work.

Quick Overview

NCTSSoS uses a thin-adapter architecture: symmetry reduction lives in a single file (src/optimization/symmetry.jl) and only takes over one job — constructing a smaller MomentProblem. Everything upstream (algebra, registries, polynomials, sparsity) and downstream (JuMP, solver, result types) is unchanged.

🔀 Three Symmetry Actions

SignedPermutation for monoid algebras, CliffordSymmetry for Pauli algebras, FermionicModePermutation for fermionic algebras

🧱 Block Diagonalization

One big PSD block → several smaller blocks via SymbolicWedderburn's representation-theoretic decomposition

🎯 Orbit Reduction

Monomials in the same orbit map to a single representative, collapsing many moment variables into few

🛡️ Fail-Fast Checks

Invariance, basis closure, and off-block vanishing are all checked — wrong but plausible answers are prevented

Core Concepts

What is a symmetry-adapted basis?

Suppose your optimization problem has a symmetry group G — relabel the operators in some prescribed way (possibly with sign flips), and the objective and constraints come back unchanged.

Then the moment matrix MB (indexed by a half-basis B of monomials) commutes with the group action. By a standard result from representation theory, there exists a change-of-basis U that block-diagonalizes M:

UT MB U = blockdiag(M(1), M(2), …)

Each block M(i) corresponds to one isotypic component — an irreducible representation of G (or a direct sum of copies of one). Because off-diagonal coupling vanishes, the single large PSD constraint MB ≽ 0 becomes several smaller independent constraints M(i) ≽ 0. Same physics, smaller SDP.

The thin-adapter design decision

There are two ways to bolt symmetry adaptation onto a moment-SOS package:

  1. Rewrite-the-package: Replace monomial coordinates with isotypic-component coordinates everywhere — polynomials, registries, simplification, sparsity.
  2. Thin adapter: Keep everything in monomial coordinates. Use the symmetry-adapted basis only to decide how to slice one PSD constraint into several, then emit smaller blocks back in monomial coordinates.

NCTSSoS deliberately takes approach (2). The output of the symmetry path is an ordinary MomentProblem — just with fewer moment variables and several small PSD blocks instead of one big one.

Trade-off: This means the feature is currently dense-only and ordinary-polynomial-only. It doesn't compose with correlative/term sparsity, state/trace polynomial optimization, or GNS reconstruction yet. But the entire non-symmetry code surface has zero regression risk.

Orbit reduction

Beyond block-diagonalization, the symmetry group identifies orbits of monomials — sets of monomials that are mapped to each other under the group action. Within each orbit, all monomials must take the same moment value (up to sign). NCTSSoS picks one canonical representative per orbit and rewrites the entire relaxation in terms of these representatives.

When an orbit has inconsistent signs (some group elements map a monomial to +representative, others to −representative), the moment is forced to zero and dropped entirely. This is the orbit-reduction step in _build_orbit_reducer.

SymmetrySpec: the user-facing entry point

Users supply generators via a SymmetrySpec, passed through the symmetry keyword of SolverConfig. The spec wraps one symmetry family at a time: 

# Monoid algebra: signed permutations on registry indices
spec = SymmetrySpec(alice_swap, bob_swap, party_swap)

# Pauli algebra: Clifford conjugation on Pauli words
spec = SymmetrySpec(CliffordSymmetry(:SWAP, 1, 2))

# Fermionic: mode permutations + sector splitting + spin adaptation
spec = SymmetrySpec(
    perm_generator;
    sector = FermionicSectorSpec(
        mode_layout = layout,
        split_parity = true,
        split_number = true,
        split_spin   = true,
        split_irrep  = true,
    ),
    spin_adaptation = FermionicSpinAdaptationSpec(mode_layout = layout),
)

Deep Dive: The Three Symmetry Actions

For MonoidAlgebra problems (NonCommutative, Projector, Unipotent), symmetries act as signed permutations on registry indices — the integers that internally identify each variable. A SignedPermutation stores (sign, target_index) pairs with sign ∈ {−1, +1}. 

# CHSH: Alice swap swaps A₁ ↔ A₂, flips sign of B₂
alice_swap = SignedPermutation(
    A[1].word[1] => A[2].word[1],        # A₁ → A₂
    A[2].word[1] => A[1].word[1],        # A₂ → A₁
    B[2].word[1] => (-1, B[2].word[1]), # B₂ → −B₂
)
# Indices not listed are fixed (B₁ stays B₁)

Internally, _act_monomial(g, mono) walks the monomial's word, maps each letter through the signed permutation, accumulates the sign, then re-sorts the word into normal form.

When to use SignedPermutation
Use SignedPermutation when your variables live in a MonoidAlgebra (NonCommutative, Projector, or Unipotent) and your symmetry is a relabeling of registry indices, possibly with sign flips. This is the right choice for Bell inequalities (CHSH, etc.) and abstract NC polynomial eigenvalue problems. For Pauli problems, use CliffordSymmetry instead — it handles Pauli phase tracking that SignedPermutation cannot.

For PauliAlgebra problems, symmetries are Clifford conjugation actions on Pauli words. Named constructors provide standard gates: 

# Hadamard on qubit 1: X₁ ↔ Z₁, Y₁ → −Y₁
h1 = CliffordSymmetry(:H, 1)

# S gate on qubit 2: X₂ → Y₂, Y₂ → −X₂
s2 = CliffordSymmetry(:S, 2)

# CNOT with control 1, target 2
cnot = CliffordSymmetry(:CNOT, 1, 2)

# SWAP qubits 1 and 2
swap = CliffordSymmetry(:SWAP, 1, 2)

# 2-site Heisenberg with SWAP symmetry
spec = SymmetrySpec(CliffordSymmetry(:SWAP, 1, 2))

Clifford actions are validated rigorously: sources must be single Pauli letters, images must be non-identity Pauli words, the map must be injective up to sign, and both commutation relations and local multiplication phases must be preserved. Custom image dictionaries are accepted too.

Under the hood, _act_monomial(g, mono) maps each Pauli letter through the image dictionary, concatenates the results, then calls simplify(PauliAlgebra, word) to restore normal form and extract the accumulated Pauli phase (+1 or −1 for Hermitian words).

Example result: A 2-site Heisenberg model H = ¼(σx₁σx₂ + σy₁σy₂ + σz₁σz₂) with SWAP symmetry produces group order 2, PSD block sizes [4, 3] (down from [7]), and objective −0.75 (exact singlet energy).

Fermionic symmetry has the richest structure. It composes three independent mechanisms:

  1. FermionicModePermutation — permutations of physical mode IDs. Unlike SignedPermutation, these are unsigned; signs come from re-normal-ordering the fermionic word (anticommutation).
  2. FermionicSectorSpec — splits the basis into blocks by quantum numbers:
    • split_parity: even/odd number of operators
    • split_number: particle number (creators − annihilators)
    • split_spin: total Sz
    • split_irrep: Abelian point-group irrep label
  3. FermionicSpinAdaptationSpec — SU(2) spin adaptation using the total S² Casimir operator, layered on top of sector splits. Within each (number, Sz) sector, it further block-diagonalizes by total spin S. 
# H₂ / STO-3G: 4 spin-orbitals, inversion symmetry
layout = FermionicModeLayout(
    Dict(1 => 1, 2 => 1, 3 => 2, 4 => 2);  # orbital_of
    spin2_of  = Dict(1 => 1, 2 => -1, 3 => 1, 4 => -1),
    irrep_of  = Dict(1 => :Ag, 2 => :B1u),
    irrep_table = AbelianIrrepTable(:Ag;
        multiply = (a, b) -> a == b ? :Ag : :B1u,
    ),
)

spec = SymmetrySpec(
    FermionicModePermutation[];       # no spatial perm needed here
    sector = FermionicSectorSpec(
        mode_layout  = layout,
        split_parity = true,
        split_number = true,
        split_spin   = true,
        split_irrep  = true,
    ),
    spin_adaptation = FermionicSpinAdaptationSpec(
        mode_layout = layout,
    ),
)

The H₂/STO-3G example in the docs shows this in action: an order-2 dense moment matrix of side ~50 is reduced to a largest PSD block of side 3, recovering the exact electronic ground-state energy.

Architecture Boundary: Who Does What

NCTSSoS (local)

  • User-facing types: SignedPermutation, CliffordSymmetry, FermionicModePermutation, SymmetrySpec, SymmetryReport
  • Active-domain computation
  • Group enumeration (BFS closure)
  • The NC-word action _act_monomial (algebra-specific simplification + Pauli phase tracking)
  • Invariance & basis-closure checks
  • Orbit reduction + zero-orbit handling
  • Final MomentProblem construction

SymbolicWedderburn (external)

  • Computing characters of the group action
  • Identifying isotypic components
  • Building the change-of-basis rows
  • Called via symmetry_adapted_basis(Float64, group, action, basis)
The thin adapter: a BySignedPermutations action type, a GroupsCore.Group wrapper (mult/inv/order tables), and an action(…, g, mono) method. That's the entire glue surface.

This boundary is clean enough that either side can be upgraded independently. NCTSSoS does not know about irreducible representations. SymbolicWedderburn does not know about Pauli phases or fermionic normal ordering.

Common Pitfalls & Gotchas

🚫 Using SignedPermutation for Pauli problems

SignedPermutation does not track Pauli phases. If you try it on PauliAlgebra variables, you'll get an ArgumentError. Use CliffordSymmetry instead — it handles the ±i phases from Pauli commutation.

🚫 Symmetry + sparsity (not supported yet)

The symmetry path requires cs_algo = NoElimination() and ts_algo = NoElimination(). Combining symmetry with correlative or term sparsity will error out. This is the main current limitation.

🚫 Non-invariant objective passes silently

If you set check_invariance = false but your objective is actually not invariant, the solver will happily return a wrong answer. This option exists for performance (the check scales with group size × problem size), but only use it when you've independently verified invariance.

🚫 Basis too small for the group action

If a group element sends a half-basis monomial outside the basis, you get a clear error. Fix: either use a larger moment_basis or remove the offending generator. The basis must be closed under the group action.

🚫 State/trace polynomials

Symmetry reduction is currently ordinary-polynomial only. State and trace polynomial problems (NCStatePolynomial) are not yet supported on the symmetry path.

⚠️ GNS reconstruction not available

The GNS reconstruction path expects a dense unreduced moment table. A symmetry-reduced solve only builds the reduced moment problem, so GNS is not available from symmetry-reduced results today.

⚠️ Mixing action families

You cannot mix SignedPermutation, CliffordSymmetry, and FermionicModePermutation generators in a single SymmetrySpec. Each spec wraps exactly one family.

Test Your Understanding

Q1: Why does NCTSSoS use a "thin adapter" instead of rewriting the package around isotypic coordinates?

Reveal answer

Because the thin adapter has zero regression risk on the non-symmetry code surface. The entire algebra hierarchy, registries, simplification rules, sparsity machinery, JuMP construction, and result types remain untouched. The symmetry feature is concentrated in one file and one boundary. The trade-off: it's dense-only today.

Q2: What two things does orbit reduction do to moment variables?

Reveal answer

(1) It replaces all monomials in the same orbit with a single canonical representative (up to ±1 sign). (2) When an orbit has inconsistent signs — some group elements map to +rep, others to −rep — the moment is forced to zero and dropped entirely.

Q3: You're optimizing a Pauli Hamiltonian that's invariant under swapping qubits 1 and 2. Which type do you use?

Reveal answer

CliffordSymmetry(:SWAP, 1, 2). Even though a SWAP is conceptually a permutation, Pauli algebra has internal phases (iXY = Z, etc.) that SignedPermutation cannot track. The library will error if you try SignedPermutation on PauliAlgebra.

Q4: Why can't symmetry reduction compose with correlative sparsity today?

Reveal answer

The symmetry path currently requires a single dense clique. With correlative sparsity, the problem would be split into multiple cliques, and the symmetry group might not preserve each clique's basis independently. Composing the two would need a more careful treatment of how orbits interact with clique boundaries.

Q5: What are the three independent mechanisms that compose in fermionic symmetry?

Reveal answer

(1) FermionicModePermutation — permutations of physical modes. (2) FermionicSectorSpec — splitting the basis by parity, particle number, Sz, and/or Abelian irrep. (3) FermionicSpinAdaptationSpec — SU(2) spin adaptation via the S² Casimir, layered on top of sectors.
